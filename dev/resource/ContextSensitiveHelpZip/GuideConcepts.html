Enterprise Service Concepts

Concepts described in this chapter include:

  • Sequences
  • Services
  • Configuration
  • Logging

Sequences

All work under the Services Package is de-constructed into specific tasks. Each task is implemented as an Apex class and is called a Service.

Logically related services are assembled by the System Configurator (or Business Analyst) into an ordered arrangement of sequential processing steps collectively called a Sequence

Services

Every service responds to an input event and emits a list of zero, one or more output events. The Services Package is designed so that output events from any given service automatically feed the input stream of the very next service in a Sequence.

The Services Package has by default a number of ready made services that can be composed sequentially so that useful work can be completed. These ready made services include:

  • Debug
  • Jump
  • Transaction
  • Publish
  • Subscribe

Additional services implementing tasks specific to your particular business requirements are easily created. Detailed instructions for building such services can be found in the Service Developers Guide

Configuration

The Services Package makes a clear distinction between:

  • what is to be performed (via declarative assembling of sequential processing steps), versus
  • how it is performed (via imperative apex code defining each service - see Service Developers Guide for further details.)

Assembling services in a useful sequence and thereby fulfilling the intent of the underlying business requirement is the domain of the business analyst and not necessarily the Apex developer. It therefore the role of the System Orchestrator / Business Analyst to configure the Services Package:

  • by arranging services in a ] Sequence using a friendly name that adequately describes the collective nature of the work to be preformed (for example ProcessInvoices)
  • by identifying tasks that as yet, have no Apex service implementation (otherwise defined as "missing services")
  • by specifying and provisioning missing services, often by securing the development expertise of a suitable Force.com developer

Logging

Salesforce prevents writing to the system log in a Production environment. As a useful substitute, the Services Package provides two logging mechanisms:

  • a DebugService which acts a no-operation service with no side effects. The Debug service will send an email containing the current event to a nominated user.
  • the Messages tab detailing the progression of any event from genesis to completion. The detailed view on any selected message displays the parent message and a related list of all child messages. In addition, the exception trace is attached to the message containing any event that fails to process correctly.
