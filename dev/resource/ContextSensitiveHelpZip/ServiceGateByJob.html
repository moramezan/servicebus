Gate By Job process

  • A related collection of messages may proceed down the same sequence in separate execution contexts.
    For most requirements, this is acceptable and desired behaviour but a need may arise for a downstream process to act on these related messages in a single transaction context.
  • An GateByJob process enables the above scenario by utilising the Job Id field as a grouping key on all related messages.
    (All messages carry the same unique value in this field when enqueued at the same time.)


Scenario

Consider a collection of 30 related events that are enqueued simultaneously.

  • Each event signals the execution of an asynchronous downstream process that for example, analyses the internal strength of a specific jet engine fan blade referenced by the event.
  • Naturally, all 30 processes must return "PASS" for the jet engine to be certified as airworthy but until the results of all 30 fan blades have completed, no positive decision can be made about the airworthiness of that jet engine.


Scenario Implementation using the Gate By Job Process

  • The enqueue method annotates all 30 "jet blade" events with the same unique Job Id.
    (Note: This is standard Service Bus functionality - a collection of simultaneously enqueued events will all receive the same Job Id)
  • Each of the 30 "jet blade" events signals the (asynchronous) execution of a custom "Blade Strength" process and this custom process will emmit a useful "PASS/FAIL" result.
  • After passing though the "Blade Strength" process, all "jet blade" events are buffered as and when they arrive by the GateByJob process
  • As soon as all 30 "jet blade" events have arrived, the GateByJob process releases a single message that contains a handle to the cumulative results of all 30 "jet blade" events. With access to this handle, all subsequent downstream processes are now equipped with a complete view of prior asynchronous events thereby enabling a valid positive assessment of the airworthiness of the jet engine in a single context.
