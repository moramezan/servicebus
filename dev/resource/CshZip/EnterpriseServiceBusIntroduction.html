Introducing Enterprise Service Bus

Enterprise Service Bus (ESB) is the only high performance messaging platform native to Salesforce. It allows systems to react to real world events. You can model sagas, asynchronous processes, and recover from errors without data loss using the drag-and-drop sequence builder.

ESB is predicated on the notion that work is broken up into processes that respond to events.

Using ESB encourages:

  • reactive, extensible and event-driven models,
  • testable components which have no knowledge of each other,
  • developers to be able to code to events via an interface.

Processes

A Process is a single piece of "encapsulated goodness" with inputs and outputs. Logic should be separated into distinct processes where:

  • the logic of a process is implemented as an Apex Class,
  • inputs to a process come in the form of exactly one message,
  • outputs of a process consist of zero, one, or many messages,

After a process receives and acts on a message, its output messages are routed to a subsequent process. Processes can split, combine, or alter data, and call Salesforce APIs.

Messages

In the real world events occur all the time. For example, the clock strikes 12PM, the fuel tank reaches empty, etc. To represent real world events in computer systems, we create a data structure called a Message which contains relevant details about the event.

The primary purpose of a message is to provide a useful mechanism to communicate between processes. The terms Message and Event are therefore used interchangeably but represent the same concept for the purposes of ths guide.

Sequences

A Sequence is an arrangement of multiple processing tasks called Steps. Each Sequence is known by a friendly name, for example 'ProcessInvoices', which should in some way indicate the nature of the work that it's processing steps will perform.

The ESB framework will advance Messages through the Steps of a Sequence in the order you specify using drag and drop. A typical sequence will contain many steps. A Sequence is like a configurable event-to-process lookup table that routes a given message to the appropriate process.
