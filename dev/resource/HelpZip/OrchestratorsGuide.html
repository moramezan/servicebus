About This Guide :
  • This guide is intended for Business Analysts and System Administrators who primarily make use of point and click actions to operate, configure, and maintain event driven applications.

  • The purpose of this guide is to:

    2. Explain basic messaging and communication concepts and terminology.

CHAPTER 01 : Configuration and Overview

You can use point and click to design and configure your event driven processing sequences.
More specifically, you can :
  • Arrange processing steps into a logical processing sequence
  • Combine two or more processing sequences by using PubSub or Branching techniques.
  • Register custom event driven process delivered via third party Salesforce packages
  • Enqueue a test message to a specific Sequence.


You can use point and click to operate and monitor your event driven processing sequences.
More specifically, you can :
  • Start and Stop the framework.
  • Restart corrected messages.
  • Monitor the system health.
  • Monitor related messages.
  • Monitor processing details within a processing sequence

CHAPTER 02 : Messaging Concepts


Messages and Events :
  • Events occur all the time. (For example: the clock strikes twelve, the fuel tank reaches empty)
  • To represent details about an event in a computer system, we use a special object called a message
  • The primary purpose of a message is to provide a useful mechanism to communicate between processes that a particular event has occurred.

Event Based Architecture :
  • Specific events are only relevant to specific processes - for this reason a broker routes event messages representing specific events to specific processes within the system.
  • Combinations of specific event messages to specific processes can be reduced to a table called a routing table. The routing table is used by the broker.
  • Breaking up work into specific processes that respond to specific event messages is called "Event Driven Architecture". (or simply "EDA" )

Command Response Architecture :
  • Most implementations on Salesforce make use of an alternative to EDA called "Command-Response Architecture".
  • Under this model, work is centrally directed from a command such as a button click or a trigger action.
  • Complexity grows in an geometric manner as possible execution paths and exception handling paths are added to the logic directly attached to the command.

Event Driven Processes :
  • As previously mentioned, events in Event Driven Architecture are represented as messages
  • A message received by a process that has previously registered it's interest in that type of event message causes the underlying Apex logic of the process to execute

Processing Sequence :
  • Event Driven Architecture allows useful work to take place on the Salesforce platform by processes that have previously registered their interest the occurrence of a specific type of event.
  • In practise this means that a process registers it's interest in events emitted from the process that immediately precedes it.
  • A register of interest containing sufficient number of processes will naturally resemble a sequence of chained processing steps
  • This register is called a Sequence and the sequential processing steps within it are called Sequence Steps

Summary :
  • Event Driven Architecture provides a framework that logically maps processing steps to an underlying business requirement.
    Emphasis is placed on narrowly defined , composable units of work , each of which responding to a specific event that has occurred.
    It provides a clean model that limits complexity in the system to just the testable apex implementation details of any given process.

  • In contrast, the Command-Response paradigm attempts to attach all possible logical code paths and associated exception handling to a single specific command.
    This creates code complexity that is not easily tested or modified.

CHAPTER 03 : Event Driven Architecture Concepts


The event driven Architecture concepts described in this chapter are:
  • Processing
  • Process
  • Routing
  • Configuration
  • Logging

Processing :
  • Work under EDA is broken up into many specific tasks
  • Each task is implemented as an Apex class and is called an EDA process.
  • Logically related processes are assembled by the System Configurator (or Business Analyst ) into sequential processing steps called a Processing Sequence

Process :
  • Every EDA process responds to an input message by emitting a list of zero, one or more output messages
  • The EDA framework is designed so that output messages from a process automatically feed the input stream of the very next process in a Processing Sequence.
  • The EDA framework has by default a number of ready made processes that can be sequenced together so that useful work can be completed.
  • These ready made processes include: Wiretap , Jump , Branch , SubSequence , Publish and Subscribe
  • Additional processes can be easily created to implement tasks specific to your particular requirement(s).
    Detailed instructions for building processes can be found here



Routing :
  • Routing is the term used to denote the marshalling of event messages between EDA processes.
  • As previously mentioned, the assembling or linking of processes in a specific order via point and click creates a sequence of Processing Steps.
  • Each Processing Step in the Sequence has a definitive position associated with it.
  • The first step in the Sequence has a position of "1", the second has a position of "2" and so forth
  • Using the ordered position of each step, the framework is able to route an event messages along it's ordained Processing Sequence

Configuration :
  • The EDA framework envisages a clear distinction between :
    • what is to be performed (via declarative point and click assembling of processes)
    • and
    • how it is performed (via imperative apex code defining each process - see Process Developer's Guide for further details. )

  • Assembling processes in a useful sequence and thereby fulfilling the intent of the underlying business requirement is the domain of the business analyst and not necessarily the Apex Developer.
    It therefore the role of the System Orchestrator / Business Analyst to configure the EDA framework :
    • by linking event driven processes together in a Processing Sequences using friendly names that adequately describe the collective nature of the work being preformed (for example "ProcessInvoices" )
    • by identifying tasks in a Sequence that have no corresponding event driven Architecture process ("missing processes")
    • by specifying and provisioning missing event driven processes by securing the development expertise of a suitable Force.com developer where necessary

Logging :

Salesforce prevents writing to the system log in a Production environment.
As a useful substitute, the EDA framework provides two logging mechanisms :
  • WireTap
    A Wiretap process acts a no-operation process and has no side effects.
    A Wiretap process will however send an email containing the current event message to a nominated user.

  • LogWriter
    A LogWriter process acts a no-operation pass through process and has no side effects.
    A LogWriter will however store the event message under a useful business identifier.
    Many LogWriter processes placed at strategic steps in a Sequence can usefully track and relate the progress of event messages using a common business identifier.

CHAPTER 04 : Framework Boundaries


External Boundary - Inbound :
  • An external event (for example "Time is 12:01" , "Stock price changed" ) is not directly consumable by the event driven framework.
  • Instead, EDA framework configurators make use of the enqueue() method to transform an external event into an event message which is injected into the EDA framework.
  • The EDA framework supports two variations of the enqueue() method, namely
    1. Enqueue(String SequenceName, Id id ) and
    2. Enqueue(String SequenceName, Blob data )

    Usage:
    The "SequenceName" parameter defines which configured processing sequence will respond to the injected message event
    Use the Enqueue(String SequenceName, Id id ) method whenever the data related to the external event is retrievable as a Salesforce object using the object's Id.
    Use the Enqueue(String SequenceName, Blob data ) method whenever the data related to the external event is a stream of data not yet persisted in Salesforce by passing in that stream in the form of a Blob.



External Boundary - Outbound :
  • Any process is free to communicate outside the framework boundary via
    • http callouts
    • the invocation of a workflow rules
    • DML operations, and/or
    • the sending of one or more email messages

Internal Boundary :
  • All Sequences have a Terminator process as their final processing step.
    (Conceptually this is very similar to a "buffer stop" at the end of a railway line)
  • A Terminator processing step is automatically inserted when the Processing Sequence is created.
  • Sequences may interact with one another via one of four standard pre-packaged processes, namely Jump, Branch, Subroutine or Publish/Subscribe - see Core processes Guide for further details.
