Process Developer's Guide

  


Intended Audience : Salesforce Apex Developer

About This Guide :
  • This guide is intended for Salesforce Apex Developers who are tasked with providing functionality not provided in the out-of-the-box ESB framework.

  • The purpose of this guide is to:
    1. explain basic plugin concepts as they pertain to the ESB framework.
    2. describe how to create a plugin that enables processing functionality not directly provided for in the ESB framework.

  • This guide excludes ESB configuration and ESB monitoring. These topics are covered in the separate Orchestrator's Guide guide.

CHAPTER 01 : Concepts : Messaging and Events


Event is a Message :
  • In the real world events occur all the time. (For example the clock strikes twelve pm, the fuel tank reaches empty, etc)
  • To represent real world events in a computer system, we often create a data structure called a message which contains relevant details about the event
  • Event and Message are therefore used interchangeably but represent the same concept for the purposes of ths guide.

Purpose of a Message :
  • The primary purpose of a message is to provide a useful mechanism to communicate between processes.

Event Based Architecture :
  • the ESB framework is based on Event Driven Architecture ("EDA")
  • SEDA is predicated on the notion that work is broken up into processes that respond to event messages.
  • the ESB framework uses a configurable event-to-process lookup table that routes a given event message to the appropriate process.

Event Messages :
  • Events in the ESB framework are represented as messages
  • An event message is handed to specific process. This in turn causes the Apex code contained within that process to execute.
    (In other words, the occurrence of a given event causes a process that has previously registered it's interest in that event to perform it's Apex logic and to respond with zero, one or many message events)
  • This is a clean model that limits complexity in the system to just the apex implementation details of any given process


CHAPTER 02 : Concepts : Process and Processing


Processing :
  • Work under the ESB framework is broken into many specific tasks
  • Each task is implemented as an Apex class and is made available to the ESB framework as a process
  • Every process executes it's task logic in a standard execution method
  • Some processes are native to the ESB framework (Gate , WireTap, Publish, etc) while others are implemented as custom plugins (the topic of this guide)
  • All ESB processes (native and plugins) adhere to a standard Apex template



Process :
  • All ESB processes contain a standard invoke() method.
  • The invoke() method takes a single input message and returns a list containing zero, one or many message(s).
    The concept can be illustrated as follows:
    List<Event> messages = process.invoke(Event message);
  • The ESB framework is designed so that the output(s) from a given process can be directly re-purposed as input(s) into immediately downstream processes.
  • The ESB framework has by default a number of pre-packaged native processes that can be sequenced together to accomplish useful work.
  • These pre-packaged processes include Branch , Gate , Jump , LogInitializer , LogWriter , Mark , Publish , Schedule , Subscribe , Subsequence , Terminate and WireTap
  • Additional processes called plugins are easily created to implement tasks specific to a particular business requirement.
  • The focus of this guide is to provide detailed guidance for the creation of these processes.

CHAPTER 04 : Development : Define the Process


Define the input event :
  • Every input event message is a loosely typed key-value map of the form Map<String,Object>
  • Of particular importance is the Id parameter.
  • The Id parameter points at the underlying data associated with the message.
  • The plugin author should be familiar with the structure of the data (SObject) pointed at by the event so that he/she can adequately retrieve and write to this object.

Define the output event(s) :
  • An output event message is takes the same shape as an input event message - that is, a loosely typed key-value map of the form Map<String,Object>
  • A process returns an output collection containing zero, one or more messages in the form List<Map<String,Object>>
    (Note ) Normally the output collection will simply contain a single output message, namely the re-purposed input message



Define the plugin activity :
  • The plugin activity represents specific logic that is unavailable under the ESB framework, either because
    1. such logic is is not represented by one of the native framework processes or
    2. such logic has not been previously created and/or registered
  • The process activity should be specific
    • An example of specific activity might be converting unix line terminators to windows line terminators
    • An example of un-specific activity might be the counting of word occurrences in a file, alphabetising the summary counts by word and finally reporting the results
  • Generally defined activities should be re-specified into separate specific activites.
    (Using the example above , one could envisage three separate activities, namel count, sort and report)
  • By tightly specifying an activity, the following two objectives are facilitated:
    1. testing : the more focused an activity , the easier it becomes to write comprehensive , reliable unit tests
    2. re-use : the more focused an activity , the easier it becomes to re-purpose the plugin in other sequences cutting down on programmer effort and the potential for bugs (less code)
  • The ESB framework surfaces the notion of process parameters via configuration and customization to accommodate variations in data applied to the same plugin behavior
    1. Customization allows a process plugin to make use of a single set of parameters without reference to the context of the sequence step containing the process.
      These parameters are defined in the Custom Settings specific to that plugin class
    2. Configuration allows an process plugin to make use of parameters having values configured that are specific to the context of the sequence step containing the process.

    (A fundamental principle of the Event Driven Architecture is a dependency reduced environment. For this reason, custom process plugins never take a dependency on the ESB framework.)



Custom Process Creation - Overview :
  • In a development org , create a new public class
  • Confirm that the class follows the expected ESB process plugin template as set out under Anatomy of a Process .
  • Build appropriate unit tests as recommended by Salesforce.

Non-Callout Example :
  • The WireTap process is a built-in NOOP framework process that sends an email to an interested user whenever it receives an event message
  • The email contains details about the sequence step hosting the WireTap process as well as the injected message parameters
  • As an alternative to the the WireTap process, we will construct a new process called ChatterTap
  • Instead of emailing an interested user , the same detail will instead be surfaced on that user's chatter feed

Activity Specification :
  • The activity specification is very similar to that of the WireTap
    (1) The activity has no side effects on processing, therefore return the untouched inbound event message as the only item in the returned output message collection
    (2) The chatter feed that will surface details about the inbound event will be the chatter feed of a a specific user as defined by the UserId field of the ChatterTap custom setting
    (3) If no UserId has been entered in the ChatterTap custom setting, use the Id of the running user context

APPENDIX I : Anatomy of a Non-Callout Process

  • Before a custom plugin can be recognised and registered in the ESB framework, the code representing the plugin must conform to an ESB template that at a minimum, consists of
    • an inner Meta class and
    • an invoke() method

  • Below is an annotated non-callout plugin called ChatterTap This plugin adheres to the required ESB template.

  • Specification : The ESB framework has a built-in WireTap process that surfaces the parameters of an inbound message via an email. The email is sent to either
    • the inbox of the running context user (default behaviour), or
    • the inbox of a nominated user as defined by a custom setting associated with the WireTap process
    This plugin will replicate the WireTap functionality.
    As an alternative to surfacing the message parameters in the form of an email, ChatterTap will instead post these parameters on a user's Chatter Feed.

  • Notes :

  • 001 : An ESB plugin is a standard Apex class. ( ESB plugins do not take a dependency on the ESB package)
  • 003 - 035 : a valid ESB plugin has two key elements, namely an inner Meta class and an invoke()method.
  • 003 - 010 : the inner Meta class exposes important properties about the ESB plugin thereby facilitating correct integration into the ESB framework.
  • 004 : the Tag is a string that that groups the ESB plugin with similar plugins.
    We recommend that you create your own category and use that for all your custom plugins.
  • 005 : the Name is a string that describes the friendly Name of the ESB plugin
  • 006 : the Description is a string that describes the functionality of the plugin and is surfaced on the list view of all ESB processes by the discovery process.
  • 007 : the Icon is a unique string that defines the icon that will appear next to the plugin when viewed on the list view of ESB processes.
    Note The full collection of possible icons can be found at the famfam website.
  • 008 : the HelpUrl is a string that describes the help page for the plugin.
    The help page does not have to be in the form of a VisualForce page - external help page reference are also valid.
  • 009 : Surfacing a link to the plugin specific custom settings on the processes list view is possible.
    Simply assign an instance of the custom setting to the CustomSetting member of the plugin's Meta inner class.
  • 013 - 035 : the invoke() method contains the Apex logic that implements the business specification.
  • 013 : an input message is injected into the invoke() method in the form of key-value collection.
  • 034 : the invoke() method returns a list of zero, one or more output messages.
    Each output message consists of a key-value parameter collection.
